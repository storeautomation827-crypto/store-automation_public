# ファイルガイド - 各ファイルの役割を理解しよう

このガイドでは、GitHub に公開するために用意された各ファイルが「何のためにあるのか」を初心者向けに説明します。

---

## ファイル一覧と役割

### 1. README.md（英語版）

#### 何のファイル？
GitHub で最初に見られるファイル。プロジェクトの「顔」です。

#### 具体的な内容
- プロジェクト全体の説明
- 何ができるのか（機能紹介）
- 使っている技術
- インストール方法
- プロジェクトの構成図

#### 何に使う？
採用担当者や他の開発者が、「このプロジェクトって何？」と疑問に思った時、最初に読むドキュメントです。

#### なぜこの内容？
- プロジェクト全体像が必要 - 何を作ったのかをすぐに理解できるように
- セットアップ方法が必要 - 実際に動かしたい人のため
- 技術スタックを明記 - どんな技術を使っているかアピール
- 英語で書く - GitHub は国際的なプラットフォームだから

#### 初心者のための説明
GitHub のプロジェクトを見たとき、まず目に入るのが README です。これが充実していないと、プロジェクトが放置されていると思われてしまいます。就活では「技術理解」と「ドキュメント作成能力」の両方が評価されるので、README はとても重要です。

---

### 2. README_JA.md（日本語版）

#### 何のファイル？
英語版 README の日本語翻訳です。ユーザーが日本語で理解できるように用意しました。

#### 具体的な内容
- README.md と同じ内容
- 日本語で説明
- 日本の就活面接で説明しやすい形

#### 何に使う？
- あなたが GitHub について説明するときの参考資料
- 就活面接で「このプロジェクトについて説明してください」と言われたときの下書き
- 親や友人に説明するときの参考

#### なぜこの内容？
README を日本語化することで：
- 採用担当者が確認しやすくなる
- 面接で説明しやすくなる
- あなた自身、プロジェクトを言語化して理解が深まる

#### 初心者のための説明
プロジェクトについて「言葉で説明する能力」は、エンジニアとして非常に重要です。README_JA を作成することで、その練習になります。

### 3. .gitignore

#### 何のファイル？
「GitHub にアップロード「してはいけない」ファイル」を指定するファイルです。

#### 具体的な内容
```
__pycache__/       ← Python がキャッシュした不要ファイル
.env               ← 環境設定（パスワードが入っていることもある）
config/users.json  ← ユーザー認証情報（絶対アップロード禁止）
data/*.db          ← データベース（実運用データ）
excel_files/       ← 店舗のプライベートデータ
.secret_key        ← セキュリティキー
```

#### 何に使う？
プロジェクトをローカルで開発するとき、秘密の情報が誤って GitHub に上がらないようにするための保険。

#### なぜこの内容？
- セキュリティ - パスワードやトークンを GitHub に上げたら漏洩する
- プライバシー - 実在する店舗名やユーザー情報を公開できない
- ビジネス保護 - 実運用データが見られると、ビジネスに支障が出る

#### 初心者のための説明
.gitignore は「社会人エンジニアの常識」です。これを理解していると、セキュリティ意識が高い開発者だと評価されます。

具体例：
- config/users.json をアップロードしてしまった → ユーザーのパスワードが流出
- data/store_automation.db をアップロード → 実際の営業データが見られてしまう
- .env に API キーが入っていた → 誰かがそのキーを悪用

こういう事故を防ぐのが .gitignore です。

---

### 4. GITHUB_SETUP_GUIDE.md

#### 何のファイル？
「このプロジェクトを GitHub にアップロードする方法」を説明したガイドです。

#### 具体的な内容
1. GitHub でリポジトリを作成する手順
2. Git ユーザー設定
3. git init → git add → git commit → git push の流れ
4. 認証方法（GitHub CLI or Personal Access Token）
5. トラブルシューティング

#### 何に使う？
このプロジェクトを本当に GitHub に公開するとき、ステップバイステップで進めるための手引き書。

#### なぜこの内容？
- 初心者だと迷う - GitHub にアップロードする方法は複数あり、初心者には難しい
- 認証が複雑 - 最近は HTTPS か SSH か Personal Access Token か、いろいろな方法がある
- エラーが起こりやすい - タイポや設定ミスでうまくいかないことが多い

#### 初心者のための説明
このガイドは「あなたが実際に GitHub に公開するときの道しるべ」です。各ステップが具体的に書かれているので、迷わず進められます。

---

### 5. COPY_INSTRUCTIONS.md

#### 何のファイル？
「どのファイルを、どこからコピーして、どこに配置するのか」を説明するガイド。

#### 具体的な内容
```
コピーすべき: app/, templates/, static/, tests/, scripts/, docs/
コピーしてはいけない: config/.secret_key, excel_files/, .env
```

#### 何に使う？
元のプロジェクト（store_automation_v1_01）から、公開用フォルダに必要なファイルを準備するときの指南書。

#### なぜこの内容？
- 何をコピーするか不明確 - プロジェクト全体をアップロードしたら、機密情報が漏洩する
- 誤り防止 - 「あ、Excel ファイルもコピーしちゃった…」みたいなミスを防ぐ
- 確認チェックリスト付き - 「ちゃんとやった？」を確認できる

#### 初心者のための説明
元のプロジェクトには、以下のような秘密がたくさん含まれています：
- 実在する店舗名（北千住店など）
- 実際の営業データ
- パスワードやトークン

これを全部 GitHub に上げてはダメです。「必要なファイルだけを選別する」というのが重要なスキルです。

### 6. config/stores.json.example と config/users.json.example

#### 何のファイル？
本当の設定ファイル（stores.json, users.json）の「ひな形」です。

#### 具体的な内容
```json
// stores.json.example
{
  "stores": [
    {
      "id": "store_001",
      "name": "Sample Store A",
      "active": true
    }
  ]
}
```

#### 何に使う？
GitHub からクローンしてきた人が、「あ、こんな形式で設定を書くんだ」と理解するための参考資料。

#### なぜこの内容？
- 本当のファイルは上げられない - 実在する店舗名が入っているから
- ひな形があると親切 - 使い手が「何を書けばいい？」と迷わない
- セキュリティと利便性のバランス - 公開情報としてサンプルは提供できる

#### 初心者のための説明
これは「説明書」のようなものです。

「このプロジェクトを使いたい人」が、「あ、stores.json ってこんな形式で書くんだ。じゃあ自分の店舗情報でこの形式を作ろう。」と思える親切な設計です。

GitHub の「ベストプラクティス」として、サンプル設定ファイルを用意することは、プロフェッショナルなプロジェクトの証です。

## ファイル構成図

```
store-automation-public/
│
├─ README.md                    ← GitHub で最初に見られる（英語）
├─ README_JA.md                 ← 日本語版（面接対策）
├─ FILE_GUIDE.md                ← このファイル（ファイル説明）
├─ GITHUB_SETUP_GUIDE.md        ← アップロード手順書
├─ COPY_INSTRUCTIONS.md         ← どのファイルをコピーするか
├─ .gitignore                   ← 秘密を守るルール
│
└─ config/                      ← 設定ファイル
   ├─ stores.json.example       ← 店舗設定のサンプル
   ├─ users.json.example        ← ユーザー設定のサンプル
   └─ .gitkeep                  ← （ディレクトリをGitに認識させるため）

※ app/, templates/, static/, tests/ などのソースコードは
  COPY_INSTRUCTIONS に従ってコピーします
```

## 各ファイルの内容が少ない理由

「え、これだけ？」と感じるかもしれませんが、これは「意図的」です。

### 理由 1: セキュリティ
実際のデータ（店舗名、ユーザー、パスワード）を公開できません。

### 理由 2: ビジネス保護
このプロジェクトが実際に動いているシステムだから、全部公開すると支障が出ます。

### 理由 3: 基盤システムのデモ
このプロジェクトは「完成品」ではなく、「今後の拡張の基礎」として設計されています。だから、コア機能（認証、Excel 操作、データベース）に焦点を当てています。

### 理由 4: 就活ポートフォリオとして最適
採用担当者は「完成度」ではなく「設計思想」「セキュリティ意識」「ドキュメント作成能力」を見ます。むしろ、適切に情報を制限できる人の方が、信頼されます。

## チェックリスト：ファイル理解度テスト

### README.md
- このプロジェクトは何か、30秒で説明できる

- インストール手順を順序立てて説明できる

### README_JA.md
- 日本語で面接官に説明できる
- アーキテクチャ図を見て説明できる

### .gitignore
- なぜ config/users.json を除外するのか説明できる
- なぜ __pycache__/ が不要なのか説明できる

### GITHUB_SETUP_GUIDE.md
- git init から git push までの流れを説明できる
- 認証方法の違い（CLI vs Token）を説明できる

- コピー対象ファイルを5つ以上言える
- なぜ excel_files/ をコピーしちゃダメなのか説明できる

## 学習の次のステップ

1. GITHUB_SETUP_GUIDE.md を読んで、本当に GitHub にアップロードしてみる
   実際に手を動かすことで、理解が深まります

2. README を少し改造してみる
   「こんなの追加したら、もっと分かりやすくなるのでは？」と考える

3. リポジトリの URL を就活面接で見せる
   「実は GitHub に公開しているプロジェクトがあります」と言える

4. フィードバックをもらう
   先生や先輩に「このファイル構成、どう？」と聞いてみる
